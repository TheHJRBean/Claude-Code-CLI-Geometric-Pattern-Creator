# Morph — gradient-stop Boundaries on one Patch, per-edge θ evaluation

The **Morph** feature (spec: `PATTERN_MORPH_SPEC.md`) spatially interpolates Figure-recipe parameters across the canvas. Grilled + signed off 2026-07-17. Three decisions worth recording:

1. **One Patch, not two patterns.** The original idea memo framed Morph as "pick a start pattern and an end pattern" from the library, which drags in a compatibility gate (same Configuration? matching tile types? conflicting strand/decoration state?). Rejected: the morph is a property of a *single* Builder Patch — the start state is the Patch's ordinary `figures`, and the user authors **Morph Boundaries** (gradient stops), each carrying its own per-Tile-type parameter values. Topology is identical by construction, so the whole compatibility branch collapses. Multiple Boundaries give "morph out and back". Configured from the Composition Phase onwards; the Gallery (read-only browser) is untouched. A future "import end-state from a save" can layer on top.

2. **θ is evaluated per edge midpoint, not per polygon.** PIC rays originate at edge midpoints, and equal θ on both sides of an edge is exactly what makes a Strand run straight through the contact point. Evaluating the interpolation parameter `t` at each **edge midpoint** (a shared edge has one midpoint, so both polygons agree on its θ) preserves that C1 continuity *by construction* — no averaging/patch-up pass. The gradient's bending is absorbed at star tips, where direction changes already live. The rejected alternatives: per-polygon centroid θ (kinks at every contact point) and per-polygon θ + post-hoc edge averaging (same result as per-edge, strictly messier). Known risk: asymmetric-θ polygons may tickle the edge-slide/pair-selection branches of `emitStarArms` — a probe suite guards this.

3. **v1 interpolates angles only, but the schema is full-capability-shaped.** Each stop stores a partial `FigureConfig` overlay per `tileTypeId`; v1 reads only `contactAngle`/`vertexContactAngle` from it, everything else is held from the start recipe. Lengths/curves interpolation lands later without a schema change. **Phase 2 — morphing the underlying tiling topology between Configurations — is explicitly deferred.**

Vocabulary (CONTEXT.md): **Morph**, **Morph Boundary**, modes **Linear** / **Radial**. Avoid "gradient" (collides with future colour gradients).

**Amendment 2026-07-18 — the start recipe is an implicit stop at the Origin.** #37 first shipped CSS-gradient stop semantics (below first stop = first stop's values; start recipe = only the base stops patch). In practice that made a single Boundary apply its angles uniformly across the whole plane and left the ordinary Composition angle sliders inert whenever a Morph was active — both reported as bugs on first real use. The field now splices an implicit stop at position 0 carrying the live start recipe (explicit stop exactly at 0 replaces it): one Boundary yields a real gradient from the Origin, and the base sliders keep driving the Origin side. `pic/morph.ts::morphFieldValue`; also matches the spec's original §Field-evaluation wording.

**Amendment 2026-07-18 (UI) — Morph requires a Frame.** "+ Add Morph" is offered only while `editor.frame` exists (a Morph pairs with a bounded artifact); an existing Morph stays editable if its Frame is removed. Disabling a Morph hides its entire canvas overlay (config retained).
