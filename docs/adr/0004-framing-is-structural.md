# Framing is a structural-only Phase; styling lives in Decoration

The **Framing** Phase produces geometry only: the **Frame**'s region and its border *geometry* (whether a border exists and where the line sits). All border *styling* — colour, width, weave/lacing — belongs to the **Decoration** Phase, alongside the gap fills and pattern line-colours Decoration already owns.

This is deliberately inconsistent with the **Composition** Phase, which *does* style what it introduces (`StrandStyle { width, color, background }`). We accept that inconsistency in exchange for a clean contract: Framing answers "what is the bounded region and where is its edge?", Decoration answers "what does everything look like?". Decoration needs a defined region to fill (see ADR-0003); making Framing structural-only means the hand-off between the two Phases is a pure piece of geometry — a region polygon plus optional border path — with no styling state to reconcile across the Phase boundary.

The decision is recorded because it is **surprising** (a reader will expect the Frame's border to be styled where the Frame is created, as strands are in Composition), it is the result of a **real trade-off** (UI convenience and Composition-consistency vs. a clean structure/style split), and it is **costly to reverse** — it defines what each Phase's UI contains and what data each Phase owns, so moving border styling between Phases later means moving both UI and config.

Consequence for v1: Frames come in two types — the **n-ring** (whole **Patch** stamps; clip-only, no gap) and parametric **Shape** Frames (square / √2 rectangle / hexagon / octagon). A Shape Frame doubles as a **completion boundary**: its edge carries **Frame nodes** one seed `edgeLength` apart and the pattern is completed *out to* it, with hard clipping only as the fallback for whatever can't be tiled. The boundary-completion tiles — and the Rays PIC emits over them — are *geometry*, so they sit on the structural side of this ADR; only their colour/weave defers to Decoration. This is what makes the structural/style split load-bearing rather than cosmetic: Framing can now generate a non-trivial amount of pattern (boundary tiles + strands), and keeping all of it unstyled until Decoration is the contract that stops styling state leaking across three Phases. Traditional arch Frames are deferred. See `CONTEXT.md` for the **Frame** definition and `project_framing_stage_idea.md` for the full design map.

## Amendment (2026-06-01): Framing is a persistent Frame overlay, not a Phase

Framing is no longer a Builder **Phase** — the **Frame** is a persistent overlay live across the Design and Composition phases and read by Decoration (see ADR-0003's amendment and the **Frame** entry in `CONTEXT.md`). The structural/style split recorded above is **unchanged**: the Frame defines the bounded region and its border *geometry*, while all border *styling* (colour, width, weave) lives in **Decoration**. Read every "Framing Phase" above as "the Frame overlay" — the decision is about the Frame, and demoting it from a Phase to an overlay does not move the structure/style boundary this ADR draws.
